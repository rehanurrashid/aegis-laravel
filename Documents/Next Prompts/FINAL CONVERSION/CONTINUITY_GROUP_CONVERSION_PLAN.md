## 🆕 STEP 0.12 — Dispute Context & Flow (Finances-first)

### Decision: disputes live inside Finances, not the sidebar

Disputes are always about a specific invoice or payout — they have no meaning outside that financial context. Putting them in the sidebar creates an orphan page with no clear entry point. The correct UX is:

- "Open Dispute" button sits right next to the invoice it targets
- The dispute list lives as a section within Finances — user never leaves the money context
- Detail view is a route (`/provider/disputes/{id}`) — already built as `DisputeDetail.vue`
- A count badge on the Finances sidebar nav item when there's an active dispute signals attention without a separate entry

### OpenDisputeModal.vue — props contract

```vue
<OpenDisputeModal
  v-model="disputeModal"
  :subject="disputeTarget"
  post-route="provider.disputes.store"
  @opened="refreshPage"
/>
```

`subject` shape: `{ type, id, amount_cents, label }`
- `type`: `'cs_invoice' | 'bp_invoice' | 'bp_payout' | 'session'`
- `id`: UUID of the invoice/payout
- `amount_cents`: integer, caps the dispute amount input
- `label`: human string shown in the modal header (e.g. "CS Invoice #INV-0042 · Marcus Webb")

On submit: `form.post(route(postRoute))` — posts `{ subject_type, subject_id, reason, amount_disputed_cents, description }`.
On success: emits `opened` (use to trigger page refresh / reload disputes list).

### Invoice eligibility rules for "Open Dispute" button

Show the button when ALL of the following are true:

| Condition | Check |
|---|---|
| Invoice status is `paid` or `sent` | `['paid','sent'].includes(inv.status)` |
| Invoice is not already `disputed` | `inv.status !== 'disputed'` |
| Invoice was issued within 60 days | `daysSince(inv.issued_at) <= 60` |
| Provider has a valid default payment method | `has_valid_default_pm` (already in props) |

When `inv.status === 'disputed'`: show a gold `Disputed` badge in the status column, disable the Pay button, and replace the action cell with a "View dispute →" link to `provider.disputes.show`.

function openDispute(inv, type) {

    id:           inv.id,

**4. Add dispute eligibility helper:**
```js
function canDispute(inv) {
  if (!['paid','sent'].includes(inv.status)) return false
  if (inv.status === 'disputed') return false
  if (!inv.issued_at) return true
  const days = (Date.now() - new Date(inv.issued_at).getTime()) / 86400000
  return days <= 60
}
```

## ⚖️ STEP 0.8 — Dispute System ✅ BUILT (backend + Vue pages)

All backend and standalone Vue pages built in batch3. Only Finances.vue integration is pending (§0.12).

### Full lifecycle recap

```
[ open ]
  Provider clicks "Dispute" on a paid/sent invoice in Finances
  → OpenDisputeModal: fills reason, amount, description
  → POST provider.disputes.store
  → DisputeService::open():
      dispute created (status: awaiting_response)
      invoice.status → disputed (blocks pay button)
      DisputeOpened event → email respondent + admin

[ awaiting_response ]
  Respondent (CS or BP) sees dispute in their own portal Finances
  → Replies via DisputeDetail.vue → POST {portal}.disputes.reply
  → status: under_review
  (If no reply after DISPUTE_RESPONDENT_REPLY_DAYS=5: stays awaiting_response, admin can still resolve)

[ under_review ]
  Admin sees dispute in admin/Disputes.vue queue
  Admin decides resolution:
    refund_full       → Stripe refund fires; invoice stays disputed
    refund_partial    → partial Stripe refund
    pay_full          → dismiss; invoice unfreezes to sent
    pay_partial       → partial payment stands; invoice unfreezes
    no_action         → dismiss; invoice unfreezes
    stripe_escalated  → beyond Aegis; party files Stripe chargeback

[ resolved ]
  Both parties emailed with outcome
  Dispute appears in "resolved" collapsed section in Finances
```
